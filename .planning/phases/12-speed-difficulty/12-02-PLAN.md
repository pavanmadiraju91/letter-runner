  <files>src/config.js, src/systems/difficulty.js</files>
  <action>
The current system uses tier-based spawnInterval (2.5s -> 1.0s over 6 levels). After Plan 01, speed is continuous but spawn interval is still discrete-tier-based. This creates a mismatch — speed ramps smoothly but obstacles still appear at tier-locked intervals.

Fix: Derive spawnInterval from current speed using a formula that creates natural density increase.

In src/systems/difficulty.js, modify getDifficultyParams():
1. Calculate spawn interval as inversely proportional to speed:
   `spawnInterval = SPEED.BASE_SPAWN_INTERVAL * (SPEED.BASE_SPEED / getCurrentSpeed())`
   At 140 px/s: 2.5s (relaxed). At 280 px/s: 1.25s (intense but fair).
   Floor at 0.8s to prevent overwhelming density.
2. Keep maxObstacles, multiplier, tallObstacles from the level system (these gate complexity, not raw difficulty).

In src/config.js:
1. Add to SPEED section: `BASE_SPAWN_INTERVAL: 2.5` (spawn interval at base speed)
2. Add to SPEED section: `MIN_SPAWN_INTERVAL: 0.8` (floor — never faster than this)
3. Remove `scrollSpeed` from each TIERS entry (it's now derived from elapsed time, not levels). Keep spawnInterval in tiers only as a fallback reference comment, but actual spawn comes from the formula.
   Actually — simpler: remove scrollSpeed from TIERS entirely since getDifficultyParams() now overrides it. The TIERS array becomes:
   ```js
   TIERS: [
     { maxObstacles: 1, multiplier: 1, tallObstacles: false },
     { maxObstacles: 2, multiplier: 1.5, tallObstacles: false },
     { maxObstacles: 2, multiplier: 1.5, tallObstacles: false },
     { maxObstacles: 2, multiplier: 2, tallObstacles: true },
     { maxObstacles: 2, multiplier: 2, tallObstacles: true },
     { maxObstacles: 3, multiplier: 3, tallObstacles: true },
   ]
   ```
4. Remove LOG_SPEED_FACTOR from DIFFICULTY (no longer needed — speed is time-based).
5. Remove DIFFICULTY.MIN_SPAWN_INTERVAL (replaced by SPEED.MIN_SPAWN_INTERVAL).
6. Update getDifficultyParams() to compute spawnInterval from the formula, applying the floor:
   ```js
   const speed = getCurrentSpeed();
   const spawnInterval = Math.max(
     SPEED.MIN_SPAWN_INTERVAL,
     SPEED.BASE_SPAWN_INTERVAL * (SPEED.BASE_SPEED / speed)
   );
   ```
7. The logarithmic overflow logic in getDifficultyParams() for levels beyond TIERS: keep it for maxObstacles and multiplier, but remove speed-related parts.

Replace the existing DEV console.table (which shows level-based params) with a time-based speed curve dump that shows the engagement profile:

```js
if (import.meta.env.DEV) {
  console.log('%c[Speed Curve]', 'color: #00ffcc; font-weight: bold');
  console.table(
    [0, 5, 10, 15, 20, 30, 40, 50, 60, 70, 80, 90].map(t => {
      const speed = Math.min(SPEED.BASE_SPEED + SPEED.ACCELERATION * t, SPEED.MAX_SPEED);
      const spawnInt = Math.max(
        SPEED.MIN_SPAWN_INTERVAL,
        SPEED.BASE_SPAWN_INTERVAL * (SPEED.BASE_SPEED / speed)
      );
      const dangerZoneMs = Math.round((getWidth() * 0.3) / speed * 1000);
      return {
        time_s: t,
        speed_px_s: Math.round(speed),
        spawn_interval_s: spawnInt.toFixed(2),
        danger_zone_ms: dangerZoneMs,
        difficulty: speed >= SPEED.MAX_SPEED ? 'CAPPED' : 
                   t < 20 ? 'easy' : t < 40 ? 'medium' : 'hard'
      };
    })
  );
}
```

This shows at a glance:
- 0-20s: easy (speed 140-180, spawn ~2.0-2.5s, danger zone >1300ms)
- 20-40s: medium (speed 180-220, spawn ~1.6-2.0s, danger zone ~1000ms)
- 40-60s: hard (speed 220-260, spawn ~1.25-1.6s, danger zone ~900ms)
- 60-70s: peak (speed 260-280, spawn ~1.25s, danger zone ~860ms)
- 70s+: CAPPED at 280 px/s

Validate the constants produce the target engagement window:

- At 70s: speed = 280 (capped). Danger zone = 240/280 = 0.86s. Hard but beatable with ~150ms reaction.

If the curve looks off, adjust ACCELERATION. Target: most deaths 30-60s.
