**In src/systems/matcher.js:**
After the for-loop that searches for a matching obstacle, if no match was found (loop completes without returning), emit a `WRONG_KEY` event: `events.emit('WRONG_KEY', { key })`. This signals that the player pressed a key with no valid target in the danger zone.

Important: Only emit WRONG_KEY if there ARE active obstacles in the danger zone (meaning the player had targets but pressed wrong). If no obstacles are in the danger zone, the keypress is simply ignored (no event, no penalty). This prevents penalizing keys pressed when the screen is empty.

**In src/systems/input.js:**
1. Import `stopLoop` and `startLoop` from '../core/game-loop.js'.
2. Import `GAME` from '../config.js'.
3. Import `events` (already imported).

4. Add a module-level `let inputLocked = false` flag and `let currentLevel = 1` variable.
   - `currentLevel` defaults to 1. Phase 5 (Difficulty Progression) will emit LEVEL_UP events — for now, subscribe to `LEVEL_UP` and update `currentLevel` from the event data. If no LEVEL_UP is ever emitted, it stays at 1 (no penalty).

5. Subscribe to `WRONG_KEY` event in `initInput()`:
   - If `currentLevel < GAME.WRONG_KEY_PENALTY_LEVEL`, do nothing (LOOP-10: no penalty at levels 1-3).
   - If `currentLevel >= GAME.WRONG_KEY_PENALTY_LEVEL`, set `inputLocked = true`, then use `setTimeout` to set `inputLocked = false` after `GAME.WRONG_KEY_DELAY * 1000` ms (LOOP-11).

8. Export a `resetInput()` function that sets `inputLocked = false` and `currentLevel = 1`. Called on game restart from main.js.

<task type="auto">
  <name>Task 2: Add visibility-change pause/resume</name>
  <files>src/systems/input.js</files>
  <action>
In src/systems/input.js, within `initInput()`:

1. Add a `visibilitychange` listener on `document`:
   ```
   document.addEventListener('visibilitychange', () => {
     if (document.hidden) {
       stopLoop();
       events.emit('GAME_PAUSED', {});
     } else {
       startLoop(/* need update/render refs */);
       events.emit('GAME_RESUMED', {});
     }
   });
   ```

   Problem: startLoop needs update/render function references which live in main.js.

   Solution: Instead of calling startLoop/stopLoop directly from input.js, emit events that main.js handles:
   - On `document.hidden` -> emit `PAUSE_REQUEST` event.
   - On `!document.hidden` -> emit `RESUME_REQUEST` event.

   Then in main.js (this plan can add it inline since main.js is already modified by plan 02):
   - Subscribe to `PAUSE_REQUEST`: call `stopLoop()`.
   - Subscribe to `RESUME_REQUEST`: call `startLoop(update, render)` and reset lastTime to avoid a huge dt spike.

   Actually, simpler approach: Since game-loop.js exposes stopLoop() and we can add a `resumeLoop()` export:
   - In game-loop.js, add `export function resumeLoop() { if (!running) { running = true; lastTime = performance.now(); requestAnimationFrame(tick); } }`.
   - In input.js: import `stopLoop, resumeLoop` from game-loop.js. On hidden -> `stopLoop()`. On visible -> `resumeLoop()`.

   This avoids needing update/render refs in input.js since resumeLoop reuses the already-stored updateFn/renderFn.

2. Add `resumeLoop` export to src/core/game-loop.js:
   ```js
   export function resumeLoop() {
     if (!running) {
       running = true;
       lastTime = performance.now();
       requestAnimationFrame(tick);
     }
   }
   ```

3. In the visibilitychange handler, also check that the game state is PLAYING before resuming (import getState and STATES from state.js). If state is GAME_OVER, don't resume — the game should stay frozen.

4. Store the visibilitychange handler reference so `destroyInput()` can remove it.
  </action>
