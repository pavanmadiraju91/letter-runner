  <action>
The existing unique letter logic in spawner.js already collects active letters into a Set and picks a random unused one. This is correct for single-spawn scenarios. However, verify and reinforce the following:

1. **Confirm the guard order is correct** — the letter selection MUST happen AFTER the max-obstacle check (already correct in Phase 2 implementation). If maxObstacles is reached, we bail before letter selection.

2. **Add a comment block explaining the uniqueness guarantee** for future maintainers:
   ```js
   // DIFF-07/DIFF-08: All visible obstacles must have unique letters.
   // We collect all active obstacle letters, then pick randomly from remaining.
   // With MAX_OBSTACLES_CAP=4, we always have 22+ available letters — collision impossible.
   ```

3. **Verify the 26-attempt loop is sufficient.** With max 4 obstacles on screen, at most 4 letters are taken from 26 — the probability of needing >5 attempts is negligible. The existing logic is fine. However, change the fallback behavior from silently bailing to using a deterministic fallback:
   ```js
   // If random didn't find one (theoretically impossible with cap=4), 
   // iterate deterministically as final safety net
   if (!letter) {
     for (let code = 65; code <= 90; code++) {
       const c = String.fromCharCode(code);
       if (!usedLetters.has(c)) {
         letter = c;
         break;
       }
     }
   }
   ```
   Keep the existing bail-out `if (!letter) return;` as the absolute last resort (26 obstacles scenario = impossible).

4. **No changes needed for multi-spawn-per-frame.** The spawner is called once per timer tick, and the timer resets. Even at minimum spawn interval (0.6s), only one obstacle spawns per tick. Multiple on-screen obstacles come from rapid successive ticks, not batch spawning. The uniqueness set collects ALL active obstacles each tick, so a newly spawned obstacle's letter is guaranteed unique against everything currently on screen.
